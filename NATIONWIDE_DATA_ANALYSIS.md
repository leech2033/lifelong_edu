# 전국 단위 데이터 설계 분석 및 보완안

## 1. 결론

현재 구조는 `전국 서비스로 확장 가능한 초안`은 맞지만, `전국 서비스용 안정 구조`로 보기는 아직 부족합니다.

핵심 이유는 다음과 같습니다.

1. 전국 17개 시도 UI는 이미 준비되어 있습니다.
2. 실제 데이터는 17개 중 일부 지역만 연결되어 있습니다.
3. 데이터 소스가 `JSON 파일`과 `페이지 내부 하드코딩 배열`로 나뉘어 있습니다.
4. 기관 목록과 기관 상세의 데이터 소스가 아직 완전히 단일화되지 않았습니다.

즉, 방향은 맞지만 현재는 `전국형 구조를 향한 과도기`입니다.

## 2. 현재 전국 데이터 상태

### 이미 데이터가 있는 지역

- 서울특별시
- 부산광역시
- 대구광역시
- 인천광역시
- 광주광역시
- 대전광역시
- 경기도
- 강원특별자치도
- 경상북도
- 경상남도
- 제주특별자치도

현재 확인 기준으로 11개 시도는 데이터가 존재합니다.

### 아직 비어 있는 지역

- 울산광역시
- 세종특별자치시
- 충청북도
- 충청남도
- 전북특별자치도
- 전라남도

즉, UI는 17개 시도를 다 보여주지만 실제 데이터 커버리지는 아직 전국 완전체가 아닙니다.

## 3. 현재 구조의 장점

### 장점 1. 전국 확장 방향 자체는 맞다

- 지역 선택 UI가 이미 전국 17개 시도 기준으로 설계되어 있습니다.
- 기관 데이터 기본 필드가 대체로 유사합니다.
- 목록형, 카드형, 지도형 등 공통 탐색 패턴이 이미 잡혀 있습니다.

### 장점 2. 원천 데이터가 삭제되지 않고 유지되고 있다

- JSON 데이터
- CSV/XLSX 원본
- 이미지 및 PDF 자산
- 지역별 하드코딩 데이터

전국 단위 서비스에서는 원천 데이터 보존이 중요하므로, 지금처럼 남겨두는 방향은 맞습니다.

### 장점 3. 지역별 수집 방식이 달라도 화면 통합이 가능하다

현재도 일부 지역은 JSON, 일부 지역은 코드 상수로 들어와 있지만, 필드만 맞추면 하나의 전국 목록으로 합칠 수 있습니다.

## 4. 현재 구조의 문제점

### 문제 1. 데이터 소스가 이원화되어 있다

- 서울/대전/부산/대구/인천은 JSON 기반
- 제주/강원/경기/경남/경북/광주는 페이지 내부 배열 기반

이 구조는 전국 서비스 운영 시 유지보수 비용이 큽니다.

### 문제 2. 페이지가 데이터 저장소 역할까지 하고 있다

특히 `client/src/pages/institutions/index.tsx`는

- 지역 데이터 보관
- 지역 설정
- 필터링
- 렌더링

을 한 파일에서 모두 처리합니다.

전국 단위로 갈수록 이 구조는 반드시 분리해야 합니다.

### 문제 3. 상세 페이지의 데이터 신뢰도가 지역별로 다를 수 있다

현재 기관 상세는 일부 데이터는 실제 기관 정보 기반, 일부는 일반화된 안내 정보 기반으로 동작합니다.

전국 서비스 기준에서는

- 목록 데이터
- 상세 데이터
- 대표 이미지
- 홈페이지
- 프로그램 데이터

의 연결 수준을 단계적으로 통일해야 합니다.

### 문제 4. 지역 커버리지 상태가 코드에 명시되어 있지 않았다

기존에는 “전국”처럼 보이지만 실제 어느 지역이 활성 데이터인지 코드상 분명하지 않았습니다.

이번 보완으로 `institutionRegions`에 `dataStatus`, `sourceType` 메타데이터를 추가했습니다.

## 5. 이번에 반영한 보완

### 보완 1. 전국 지역 메타데이터 명시화

파일:

- `client/src/data/institutionRegions.ts`

추가 내용:

- `dataStatus`
- `sourceType`

이제 각 지역이

- 실제 데이터 연결 상태인지
- 향후 수집 예정인지
- JSON 기반인지
- 인라인 데이터 기반인지

를 코드 레벨에서 구분할 수 있습니다.

### 보완 2. 기관 상세의 전국 데이터 완충 처리

파일:

- `client/src/pages/institutions/detail.tsx`

일부 지역의 상세 데이터가 아직 완전히 연결되지 않아도, 상세 화면이 바로 깨지지 않고 “정보 준비중” 형태로 안전하게 동작하도록 보완했습니다.

이건 삭제가 아니라, 전국 데이터 통합 과정에서 서비스가 끊기지 않게 하는 안전장치입니다.

### 보완 3. 기관 타입과 카탈로그 구조 분리

파일:

- `client/src/types/institution.ts`
- `client/src/data/institutionCatalog.ts`

이제 기관 데이터의 공통 타입과 상세 변환 로직을 페이지 밖으로 빼기 시작했습니다.

이 작업은 향후 전국 데이터 통합의 출발점입니다.

## 6. 전국 단위 서비스 기준 권장 목표 구조

권장 구조는 아래와 같습니다.

### 1단계. 지역별 원천 데이터 보존

- `raw/` 또는 기존 `attached_assets/`
- 수집 원본 CSV/XLSX/PDF

### 2단계. 지역별 정규화 데이터

- `client/src/data/institutions/<region>.json`
또는
- `server/data/institutions/<region>.json`

여기서는 전국 모든 지역이 같은 스키마를 가져야 합니다.

### 3단계. 전국 통합 인덱스

- 전체 기관 목록
- 지역별 개수
- 검색용 키워드 필드
- 상세 연결용 `id`

### 4단계. 상세 데이터 계층

- 기관 기본 정보
- 홈페이지
- 대표 이미지
- 프로그램
- 운영 메타데이터

목록용 데이터와 상세용 데이터를 분리하면 전국 서비스 운영이 훨씬 쉬워집니다.

## 7. 전국 서비스 관점에서 가장 중요한 다음 보완

### 우선순위 1

`institutions/index.tsx` 내부의 지역별 배열 데이터를 페이지 밖으로 모두 이동

이게 가장 중요합니다.

### 우선순위 2

전국 기관 공통 스키마 확정

추천 필드:

- id
- name
- region
- detail_region
- category
- address
- phone
- homepage
- tags
- status
- image
- sourceRegion
- sourceType
- updatedAt

### 우선순위 3

지역별 데이터 수집 상태 대시보드화

예:

- 수집 완료
- 정규화 완료
- 상세 연결 완료
- 이미지 연결 완료

### 우선순위 4

상세 페이지를 전국 데이터 전체 기준으로 연결

현재는 1차 보완만 되어 있으므로, 다음에는 실제 모든 지역 데이터가 `institutionCatalog`를 통해 조회되도록 완전 통합해야 합니다.

## 8. 최종 판단

현재 구조는 `전국 서비스로 가는 방향성은 맞다`고 볼 수 있습니다.

하지만 아직은

- 전국 UI
- 부분적 전국 데이터
- 혼합된 데이터 저장 방식

이 공존하는 단계입니다.

따라서 지금 필요한 건 데이터 삭제가 아니라,

`전국 원천 데이터를 유지한 채, 지역별 데이터를 동일 스키마로 외부화하고, 목록/상세/검색의 단일 데이터 계층을 만드는 것`

입니다.

이 방향으로 가면 전국 단위 서비스 설계가 안정적으로 완성됩니다.
