# 외부 호스팅 전 체크리스트

## 결론

- 현재 프로젝트는 외부 호스팅이 가능하다.
- 다만 `수정 없이 즉시 배포`보다는 `배포 안정화 목적의 최소 수정 후 배포`가 적절하다.
- 지금 단계에서는 기능 확장보다 배포 안정성과 공개 범위 정리가 우선이다.

## 1. 배포 방식 결정

- 정적 배포로 갈지, Node 서버 포함 배포로 갈지 먼저 고정한다.
- 현재 구조는 Express 서버를 포함하지만 실제 서비스는 프런트 중심 포털에 가깝다.
- 우선 추천:
  - 1차 공개: 정적 배포
  - 2차 확장: 서버/API가 필요해질 때 Node 호스팅 전환

### 정적 배포가 맞는 경우

- API가 사실상 비어 있다.
- 공개 목적이 기관/강좌/프로그램 탐색 포털이다.
- Firebase Hosting, Vercel, Netlify 같은 정적 호스팅이 목표다.

### 서버 포함 배포가 맞는 경우

- 이후 로그인, 관리자 기능, API, DB 연결을 바로 붙일 계획이다.
- App Hosting, Cloud Run, Render, Railway 같은 Node 호스팅이 목표다.

## 2. 코드 수정이 필요한 항목

### 꼭 수정하는 것이 좋은 항목

- 내부 전용 페이지 공개 여부 정리
  - `/internal/institution-quality`
  - 외부 공개가 불필요하면 링크 제거 또는 접근 제한 검토
- 메타 정보 정리
  - 페이지 title
  - description
  - open graph 이미지
  - favicon
- 공개용 문구 검수
  - 메인 화면
  - 기관 목록
  - 기관 상세
  - 404
  - 대시보드/내부 화면 링크
- 프로덕션 빌드 확인
  - `npm.cmd run build`
  - `npm.cmd start`
- 배포용 URL 정책 확인
  - 루트 경로
  - 내부 경로
  - SPA 라우팅 fallback

### 지금은 굳이 하지 않아도 되는 항목

- DB/API 대규모 도입
- 관리자 기능 고도화
- 전국 데이터 전면 재수집
- UI 전면 개편

## 3. 데이터/콘텐츠 점검

- 전국 기관 카탈로그는 단일 소스로 정리되어 있다.
- 현재 검수 리포트 기준 `needs_review` 항목은 0건이다.
- 다만 원천 데이터 품질이 완전히 정제된 것은 아니므로 공개 화면 핵심 구간은 수동 확인이 필요하다.

### 배포 전 수동 확인 권장 경로

- `/`
- `/institutions?region=all&view=list`
- `/institutions/detail/<샘플 ID>`
- `/online`
- `/local`
- `/internal/institution-quality`

## 4. 운영 리스크

### 공개 시 바로 문제될 수 있는 항목

- 내부 전용 페이지가 외부에 그대로 노출되는 것
- 대시보드가 인증 없이 열리는 것
- 일부 한글 문구가 화면에 남아 있는 것
- 배포 대상이 정적인데 서버 구조를 그대로 유지해 설정이 복잡해지는 것

### 지금 허용 가능한 항목

- 실제 API가 없는 목업형 대시보드
- 정적 데이터 기반 상세 설명
- 후속 정제 예정인 일부 데이터 필드

## 5. 배포 직전 실행 체크

1. `npm.cmd install`
2. `npm.cmd run check`
3. `npm.cmd run build`
4. `npm.cmd start`
5. 브라우저에서 주요 경로 확인
6. 내부 페이지 공개 여부 최종 결정
7. GitHub 업로드
8. 호스팅 배포

## 6. 추천 판단

- 현재 시점에서는 `배포 안정화용 최소 수정 후 외부 호스팅`이 가장 적절하다.
- 즉, 소스를 아예 손대지 않는 것은 비추천이다.
- 하지만 구조를 크게 갈아엎는 리팩토링도 지금은 비효율적이다.
- 목표는 `보여줄 수 있는 수준으로 정리한 뒤 빠르게 배포`다.
